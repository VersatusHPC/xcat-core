# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

xCAT is a cluster deployment and administration toolkit. This repo is a Perl-heavy monorepo where each top-level `xCAT*` directory is an independent package that builds to its own RPM (and often its own `.deb`). Each subproject carries its own spec file (e.g. `xCAT-server/xCAT-server.spec`) plus a `debian/` directory for Ubuntu builds.

The main runtime split:

- **`perl-xCAT/xCAT/`** — shared Perl library (`xCAT::Table`, `xCAT::NetworkUtils`, `xCAT::NodeRange`, …) consumed by every other subproject. Changes here ripple through the whole system.
- **`xCAT/`** — management-node package (postscripts, templates, prescripts, config).
- **`xCATsn/`** — service-node variant of the above.
- **`xCAT-server/lib/xcat/plugins/`** — the plugin system dispatched by `xcatd` (the xCAT daemon in `xCAT-server/sbin/xcatd`). Adding a new command usually means adding a plugin here.
- **`xCAT-client/`** — user-facing CLIs (`nodels`, `rpower`, …).
- **`xCAT-genesis-builder/` + `xCAT-genesis-scripts/`** — tooling to build the diskless initrd (`xCAT-genesis-base`) used for node discovery/netboot.
- **`xCAT-probe/`** has a quirk: `buildcore.sh`/`makerpm` copy `NetworkUtils.pm`, `GlobalDef.pm`, `ServiceNodeUtils.pm` from `perl-xCAT/xCAT/` into `xCAT-probe/lib/perl/xCAT/` at build time so the probe RPM stays self-contained. Don't edit those copies — edit the originals in `perl-xCAT/`.

Version/release are read from plain files at the repo root: `Version`, `Release`, `Gitinfo`. `Gitinfo` is generated by the build scripts; if it's missing, the build scripts (notably `buildrpms.pl`) will fail until it's produced.

## Building

Two generations of build scripts coexist — prefer the newer `buildrpms.pl` path.

### Primary: `buildrpms.pl` (mock-based, used by current CI/test flow)

```bash
# one-time host setup (installs mock, nginx, podman, etc.)
./buildrpms.pl --install_deps

# build every xCAT component for every supported target
./buildrpms.pl

# narrow it down
./buildrpms.pl --target rhel+epel-9-x86_64 --package perl-xCAT --force
```

Targets default to `rhel+epel-{8,9,10}-x86_64`. The script drives `mock`, writes RPMs to `dist/<target>/rpms`, runs `createrepo`, and regenerates `/etc/nginx/conf.d/xcat-repos.conf` so the repo is served on port 8080. `--force` overrides the skip-if-already-built behaviour. Runtime dependencies live in `../xcat-dep/el{8,9,10}/x86_64/` — that tree must exist (extracted from the xcat-dep tarball) for the test containers, but is *not* needed just to build.

Important nginx caveat: the script does **not** configure nginx's main `listen` directive. Edit `/etc/nginx/nginx.conf` manually to listen on 8080 before the first run.

### Single component: `makerpm <component> [arch]`

Builds one RPM using `rpmbuild` directly (no mock). Used by `buildcore.sh` and still handy for quick iteration on one package. `makerpm-genesisbuilder` is the equivalent for `xCAT-genesis-builder`.

### Genesis base RPM

```bash
cd xCAT-genesis-builder && ./buildrpm
```

Must run on a Red Hat-family host matching the target arch; builds `xCAT-genesis-base-<arch>` into `~/rpmbuild/RPMS/noarch/`.

### Debian/Ubuntu

`build-ubunturepo` is the Ubuntu equivalent of `buildcore.sh`; `build-python-deb` builds the Python-specific debs. The GitHub Actions workflow uses these via `github_action_xcat_test.pl`.

### Legacy: `buildcore.sh`

Full build + upload to xcat.org. Kept for release engineering; not the day-to-day path. Reads env-style `attr=value` args (`PROMOTE=1`, `BUILDALL=1`, `EMBED=zvm`, …).

## CI — CRITICAL REQUIREMENTS

**Every "Build & Test" CI job MUST both build packages AND run the full test harness.** This applies to ALL targets — EL9, EL10, Ubuntu-22.04, Ubuntu-24.04 on both x86_64 and ppc64le.

The test harness validates:
- **xcatd running** and accepting client connections
- **Unit tests**: `xcattest -s ci_test` (235 test cases)
- **make\* smoke tests** (BATS): makehosts, makedns, makedhcp, makeknownhosts, makeconservercf, makentp
- **Image generation**: copycds, genimage, packimage
- **Stateless and stateful deployment** to compute node VMs
- **Boot method coverage**: xnba (BIOS), grub2 (UEFI) for x86_64; petitboot, grub2 for ppc64le

A "Build & Test" job that only builds without running the harness is **incomplete and must be fixed**.

See `ci/CI-SPEC.md` for the full specification, requirements (R1-R7), and decision log.

## Testing

### Local install/validate in a container (the main manual test loop)

```bash
# after buildrpms.pl has produced a repo and nginx is serving it on :8080
test/scripts/setuptesthost.pl --setup_container --target rhel+epel-10-x86_64
podman exec -it xcattest-el10 scripts/testxcat.pl --all
```

`setuptesthost.pl` builds `test/Containerfile` (a systemd-capable Rocky Linux image) and creates a container named `xcattest-el{8,9,10}`. `testxcat.pl` has phase flags — `--setup_repos`, `--install`, `--uninstall`, `--reinstall`, `--validate`, or `--all` — so you can rerun just one step. It points the container at the host's nginx via `host.containers.internal`. See `test/README.md` for the full checklist.

## Conventions worth knowing

- When editing shared Perl in `perl-xCAT/xCAT/*.pm`, remember that `xCAT-probe` ships copies of three of those files; the copies are regenerated by `makerpm`/`buildcore.sh`, so don't commit hand-edits inside `xCAT-probe/lib/perl/xCAT/`.
- Subproject `.spec` files are the source of truth for what each RPM ships. Adding a new file under e.g. `xCAT-server/` usually requires adding it to `xCAT-server/xCAT-server.spec`'s `%files` section and to the matching `debian/*.install` entry.
- All build scripts are written to be **idempotent** — they skip work already done unless passed `--force`. Preserve that property when modifying them.
- License header on new files: EPL (`# IBM(c) <year> EPL license http://www.eclipse.org/legal/epl-v10.html`) — match the header style of neighbouring files in the same subproject.
- Documentation lives in `docs/` (reStructuredText, built with `make html`, hosted at readthedocs). README-level user-facing docs should go there, not in the repo root.
